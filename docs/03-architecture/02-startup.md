## 3. 위협 모델링
### 3.1 식별된 위협 목록

| 위협 | 위협 설명 |
|------|-----------|
| S3 직접 접근 | 사용자가 S3 URL로 직접 접근을 시도합니다. |
| ALB 직접 우회 공격 | CloudFront를 거치지 않고 ALB를 직접 호출합니다. |
| DB 외부 접근 | 외부에서 RDS에 직접 연결을 시도합니다. |
| EC2 직접 접근 (SSH) | 외부에서 EC2 서버에 SSH 접속을 시도합니다. |
| IAM 권한 남용 (기본 상태) | 평소 admin 권한이 상시 노출되어 크리덴셜 탈취 시 즉시 악용됩니다. |
| 로그 삭제 시도 | 공격자가 침입 후 CloudTrail 로그를 삭제하여 흔적을 제거합니다. |
| 평문 통신 공격 (MITM, Sniffing) | HTTPS가 아닌 HTTP 구간에서 중간자 공격이나 패킷 가로채기가 발생합니다. |
| S3 데이터 탈취 (외부) | 버킷이 퍼블릭으로 열려 외부에서 익명 접근이 가능해집니다. |
| Secrets 노출 (코드) | DB 비밀번호, API 키 등이 코드나 환경 변수에 평문으로 포함됩니다. |
| L3/L4 DDoS | CloudFront 앞단에 대량의 네트워크 계층 트래픽 공격이 발생합니다. |
| L7 공격 (SQLi, XSS 등) | WAF 없이 애플리케이션 계층 공격이 들어옵니다. |
| 대규모 HTTP DDoS (L7 Flood) | L7 HTTP Flood를 자동 차단하는 체계가 없습니다. |
| Zero-day / APT 공격 | 정교한 이상행위 탐지 체계 없이 고급 공격이 발생합니다. |
| 네트워크 레벨 이상 탐지 부족 | VPC Flow Logs 미사용으로 L3/L4 포렌식이 어렵습니다. |
| EC2 Public Subnet 노출 | App Server가 Public Subnet에 있어 Private 구조보다 노출면이 존재합니다. |
| 단일 AZ 장애 | AZ 장애 시 서비스가 중단됩니다. |
| 내부자 악의적 행동 | 정당한 권한을 가진 내부자의 고의적 악성 행위를 완전히 차단할 수 없습니다. |
| IAM User 기반 구조 한계 | SSO 미사용으로 중앙 집중형 인증과 계정 수명주기 관리에 한계가 있습니다. |
| ALB 주소 노출 가능성 | CloudFront를 앞에 두더라도 ALB 주소 자체를 완전히 숨기기 어렵습니다. |
| 로그 기반 대응 한계 | 이상행위를 자동 격리/차단하는 체계 없이 탐지와 알림 중심으로 운영합니다. |

### 3.2 아키텍처에서 대응된 위협

| 위협 | 위협 설명 | 방어 방법 | 방어 방법 상세 설명 |
| --- | --- | --- | --- |
| S3 직접 접근 | 사용자가 S3 URL로 직접 접근 시도 | CloudFront + OAC + bucket policy | S3 버킷이 CloudFront 배포만 신뢰하도록 제한되어 있어 직접 URL 요청은 거부된다. |
| ALB 직접 우회 공격 | CloudFront 거치지 않고 ALB 직접 호출 | SG + CloudFront prefix + custom header | CloudFront IP 대역과 커스텀 헤더를 동시에 검사하므로 우회 요청이 ALB에서 차단된다. |
| DB 외부 접근 | RDS 직접 공격 | Private subnet + SG (EC2만 허용) | DB는 퍼블릭 엔드포인트가 없고 EC2 Security Group만 허용되어 외부 직접 연결이 불가능하다. |
| EC2 직접 접근 (SSH) | 외부에서 서버 접속 시도 | SSH 차단 + SSM | 22번 포트를 열지 않고 SSM으로만 관리 접속을 허용하므로 인터넷에서 직접 로그인할 수 없다. |
| IAM 권한 남용 (기본 상태) | 평소 admin 권한 노출 | AssumeRole + ReadOnly 기본 | 관리자 권한이 상시 부여되지 않고 필요 시 일시적으로만 획득되므로 기본 노출면이 줄어든다. |
| 로그 삭제 시도 | 공격자가 흔적 제거 | Permission Boundary로 CloudTrail 보호 | 관리자 역할이더라도 CloudTrail 중지·삭제 권한은 제한되어 로그 은닉이 어렵다. |
| 평문 통신 공격 | MITM, sniffing | HTTPS 강제 + S3 SecureTransport | HTTP 요청은 차단되며 저장소 접근도 TLS가 아닌 경우 거부된다. |
| S3 데이터 탈취 (외부) | public bucket 문제 | bucket policy + IAM role 제한 | 버킷 공개를 막고 특정 역할만 접근 가능하게 제한해 외부 익명 접근이 불가능하다. |
| Secrets 노출 (코드) | 코드에 DB 비밀번호 포함 | Secrets Manager + IAM role | 애플리케이션이 실행 시점에만 필요한 비밀을 읽으므로 코드와 저장소에 비밀번호가 남지 않는다. |
| CloudFront 기반 DDoS (L3/L4) | 대량 트래픽 공격 | AWS Shield Standard | CloudFront 앞단에서 AWS가 기본 제공하는 네트워크 계층 DDoS 완화 기능이 적용된다. |

### 3.3 수용(감수)한 위협 및 근거

| 위협 | 현재 한계 | 이 규모에서 감수하는 이유 |
| --- | --- | --- |
| **L7 공격 (SQLi, XSS 등)** | WAF를 사용하지 않으므로 관리형 룰 기반의 자동 차단은 없다. 현재는 입력값 검증, 인증 로직, 헤더 검증, ALB 설정으로 방어한다. | 현재 서비스는 MAU 100 수준의 소규모 서비스이고, 대규모 공개 서비스처럼 지속적인 웹 공격의 주요 표적이 될 가능성이 상대적으로 낮다고 판단했다. 또한 WAF는 단순히 붙인다고 끝나는 서비스가 아니라 룰 관리와 오탐 대응이 필요하므로, 지금 단계에서는 애플리케이션 레벨 방어를 우선 강화하는 것이 더 현실적이라고 보았다. |
| **대규모 HTTP DDoS** | CloudFront와 Shield Standard로 L3/L4 수준의 분산 공격 완화는 가능하지만, L7 HTTP Flood를 정교하게 자동 차단하는 체계는 없다. | 현재 구조에서는 CloudFront가 엣지에서 정적 트래픽을 흡수하고 ALB 및 앱 로그로 이상 요청을 탐지할 수 있다. MAU 100 규모의 서비스가 대규모 HTTP Flood의 우선 표적이 될 가능성은 높지 않다고 보고, 우선은 탐지와 수동 대응으로 감수한다. 향후 서비스 규모가 커지면 WAF Rate-based rule 도입을 다음 단계로 본다. |
| **Zero-day / 고급 공격 (APT)** | GuardDuty나 고급 위협 탐지 체계를 두지 않았기 때문에 정교한 이상행위 탐지는 제한된다. | 현재 팀은 4인 규모이고, 고급 탐지 서비스를 붙이더라도 이를 분석하고 대응할 인력이 부족하다. 즉, 탐지 자체보다 대응 가능성이 더 중요한 상황이다. 따라서 지금은 계정 오용, 보안 설정 변경, 기본적인 웹 공격 같은 현실적인 위협을 우선 대상으로 삼고, APT 수준 위협은 현재 단계에서 감수하는 것이 합리적이라고 판단했다. |
| **네트워크 레벨 이상 탐지 부족** | VPC Flow Logs를 사용하지 않으므로 L3/L4 수준의 네트워크 포렌식과 세밀한 트래픽 분석은 어렵다. | 현재 아키텍처는 외부 진입점이 사실상 ALB 하나로 제한되어 있고, HTTP 요청은 ALB Access Log에서, AWS API 행위는 CloudTrail에서 볼 수 있다. 즉, 현재 규모와 구조에서는 핵심 위협을 상위 로그만으로도 상당 부분 추적할 수 있다고 판단했다. 대응 역량이 제한된 상황에서 저수준 로그까지 전부 쌓는 것은 효율이 낮다고 보았다. |
| **EC2 Public 노출 위험** | App Server가 Public Subnet에 있어 네트워크 관점에서는 더 보수적인 Private 구조보다 노출 면이 있다. | 다만 실제 접근은 ALB Security Group과 SSM으로 제한되어 있고, SSH 포트도 열지 않는다. 즉, Public에 있다고 해서 누구나 직접 접속 가능한 구조는 아니다. 현재 규모에서는 NAT Gateway, Bastion Host, 별도 관리망까지 두는 것보다 접근 경로를 강하게 제한한 단순 구조가 운영 실수 가능성까지 포함하면 더 현실적이라고 판단했다. |
| **단일 AZ 장애** | AZ 장애가 발생하면 서비스가 중단될 수 있다. | 이 위험은 기밀성이나 무결성 문제보다는 가용성 문제에 가깝다. 현재는 MAU 100 수준의 소규모 서비스이고, 업무 특성상 수십 분에서 수 시간 수준의 다운타임은 감수 가능한 수준이라고 판단했다. 현재 단계에서는 Multi-AZ로 인한 비용 증가와 운영 복잡도보다 단일 AZ의 단순성이 더 크다고 보았다. |
| **내부자 악의적 행동** | Permission Boundary, CloudTrail, MFA, AssumeRole로 완화는 가능하지만, 정당한 권한을 가진 내부자의 고의적 악성 행위를 완전히 차단할 수는 없다. | 내부자 위협은 어느 환경에서도 완전 차단이 어렵다. 현재 4인 소규모 팀에서는 복잡한 통제 체계를 도입하기보다, 관리자 권한 상시 노출을 줄이고 모든 권한 상승과 주요 변경을 추적하는 구조가 더 현실적이다. 따라서 현재는 예방과 억제, 사후 추적 중심으로 대응하고 완전 차단은 감수한다. |
| **IAM User 기반 구조** | SSO와 Identity Center를 쓰지 않기 때문에 중앙 집중형 인증과 계정 수명주기 관리 측면에서는 한계가 있다. | 현재는 단일 AWS 계정과 4명의 사용자만 관리하면 되는 구조라서 IAM User 기반 운영이 가능하다고 보았다. 지금 단계에서 Organizations, SSO, 다계정 구조까지 도입하면 관리 복잡도가 오히려 과도하게 증가할 수 있다. 대신 ReadOnly 기본, AssumeRole, MFA, CloudTrail 추적으로 위험을 낮추는 방향을 택했다. |
| **ALB 직접 IP 또는 도메인 노출 가능성** | CloudFront를 앞단에 두더라도 ALB 주소 자체를 완전히 숨기는 것은 어렵다. | 다만 ALB Security Group에서 CloudFront Prefix List만 허용하고, 커스텀 헤더 검증까지 적용하면 직접 접근 시도 대부분을 차단할 수 있다. 즉, 완전 은닉은 아니지만 우회가 실제로 성공하기 어렵게 만드는 구조다. 현재 규모에서는 완전 은닉보다 우회 불가 통제를 더 현실적인 목표로 두었다. |
| **로그 기반 대응 한계** | 현재 구조는 탐지와 알림 중심이며, 이상행위를 자동으로 격리하거나 실시간 차단하는 체계는 없다. | 소규모 팀에서는 자동 차단을 잘못 설계하면 오탐으로 정상 서비스에 더 큰 영향을 줄 수 있다. 따라서 지금은 CloudTrail, CloudWatch, ALB Log, 앱 로그를 기반으로 핵심 이상징후를 빠르게 인지하고 사람이 대응하는 구조를 택했다. 현재 인력과 운영 수준을 고려하면 이것이 더 안정적이라고 판단했다. |
